TraceBullet, explained for an interview demo

Use this as your speaking map: product problem first, deterministic investigation second, code path third, and tradeoffs last.

The 20-second answer

TraceBullet is a local command-line incident investigation tool. You give it a Sentry Issue ID, it checks recently merged GitHub pull requests for the same service, adds nearby Slack context when it can, and prints the suspected causing PR with evidence.

The important interview phrase: TraceBullet does not guess. The MVP uses deterministic rules before any optional narrative layer.

The end-to-end flow

1 Input

A developer runs an investigation with a Sentry Issue ID.

2 Data source

The CLI loads either local prototype data or Coral sandbox data.

3 Matching

The core algorithm requires a Service Match and Time Match.

4 Evidence

Slack context is attached only when a pre-incident marker exists.

5 Output

The CLI prints a human report or JSON Machine Report.

The demo command

node src/cli.ts investigate SENTRY-TB-1001
node src/cli.ts investigate SENTRY-TB-1001 --json

The first command is the clean human demo. The second shows that the same facts are available as structured JSON for tests and future UI work.

How the code is organized

src/cli.ts Parses investigate, reads flags, chooses source, measures runtime, and formats output.
src/investigation.ts Holds the core rules: 30-minute window, exact service tag match, closest prior candidate wins.
src/localPrototypeData.ts Contains repeatable fake Sentry, GitHub, and Slack records for a safe local demo.
src/coralSandboxData.ts Builds and runs Coral SQL against sandbox Sentry, GitHub, and Slack sources, then normalizes rows into the same data shape.
test/investigate-cli.test.js Documents expected behavior: success, JSON output, missing Slack, no-match cases, and Coral configuration errors.
prototypes/... Captures a future UI direction: one action per screen, not a dense dashboard in the MVP.

The matching rule

A pull request becomes a candidate only if it has the same service tag as the Sentry issue and was merged during the 30 minutes before the issue first appeared. If more than one candidate exists, TraceBullet chooses the one merged closest to the issue time.

Slack context is useful, but not required. That is a deliberate design choice: chat can support the report, but it should not decide whether a PR is a candidate.

What to say about architecture

  • The project starts with a CLI because the riskiest part is the investigation logic, not the UI.
  • Local prototype data makes the demo repeatable without touching production systems.
  • Coral sandbox data proves the same flow can read real external sources while staying bounded to safe demo accounts.
  • JSON output keeps the CLI useful for automated tests and later UI rendering.

Self-check

Why does TraceBullet avoid saying root cause?

Because the evidence is strong but bounded. The tool identifies a Suspected Causing PR, not a proven root cause.

What are the two required matches?

Service Match and Time Match. Slack context is supporting evidence, not a requirement.

Where would you start reading the code?

Start in src/cli.ts, then follow the call into investigateSentryIssue in src/investigation.ts.
